# 附录 A3 - 架构决策记录（ADR）模板

> Architecture Decision Records — 记录重大技术决策的上下文、方案对比和最终选择

## ADR 模板

```markdown
# ADR-[编号]: [决策标题]

## 状态
[提议 | 已采纳 | 已废弃 | 已替代（被 ADR-xxx 替代）]

## 日期
[YYYY-MM-DD]

## 决策者
[参与决策的人员]

## 上下文
描述面临的问题或需要做出决策的背景：
- 当前状况是什么？
- 为什么需要做这个决策？
- 有哪些约束条件？（技术、团队、时间、成本）

## 备选方案

### 方案 A: [名称]
- **描述**: 简要说明方案内容
- **优势**:
  - 优势 1
  - 优势 2
- **劣势**:
  - 劣势 1
  - 劣势 2
- **成本/风险**: 评估

### 方案 B: [名称]
- **描述**: ...
- **优势**: ...
- **劣势**: ...
- **成本/风险**: ...

### 方案 C: [名称]（如有）
- ...

## 决策
选择 **方案 [X]**。

## 理由
解释为什么选择这个方案：
1. 关键理由 1
2. 关键理由 2
3. 关键理由 3

## 影响
- 对架构的影响
- 对团队的影响（学习成本、招聘等）
- 对后续决策的影响
- 需要注意的风险和缓解措施

## AI 辅助记录（如适用）
- **工具**: [使用的 AI 工具]
- **Prompt 摘要**: [用于辅助决策分析的 Prompt]
- **AI 贡献**: [AI 在决策过程中提供了什么帮助]
- **人工判断**: [最终决策中人工考量了哪些 AI 未覆盖的因素]
```

## 案例：慢病管家项目 ADR 记录

### ADR-001: 前端框架选型

**状态**: 已采纳
**日期**: [项目启动日期]
**决策者**: 技术负责人、前端负责人

**上下文**:
慢病管家需要开发微信小程序，团队有 React 和 Vue 经验。未来可能扩展到支付宝小程序和 H5。项目大量使用 AI 辅助代码生成，需要考虑 AI 对不同框架的生成质量。

**备选方案**:
- A: 原生微信小程序
- B: Taro 3.x (React + TypeScript)
- C: uni-app (Vue 3 + TypeScript)

**决策**: 选择 **方案 B — Taro (React + TypeScript)**

**理由**:
1. React + TypeScript 在 AI 训练数据中占比最高，代码生成质量最优
2. TypeScript strict 模式对医疗数据模型的类型约束至关重要
3. 跨端能力为后续 H5 医护端预留空间
4. 团队 React 经验更丰富

**影响**:
- 需要学习 Taro 特有的 API 和限制
- 小程序包体积略大于原生（可通过分包优化）
- 部分微信原生能力需要通过 Taro 插件桥接

---

### ADR-002: 后端框架与数据库选型

**状态**: 已采纳
**日期**: [项目启动日期]
**决策者**: 技术负责人、后端负责人

**上下文**:
需要支持模块化架构，未来可演进到微服务。医疗数据需要强一致性和事务支持。团队希望技术栈简洁、学习成本低，同时保证 AI 代码生成质量。

**备选方案**:
- A: Flask + SQLAlchemy + PostgreSQL
- B: FastAPI + SQLAlchemy + PostgreSQL
- C: Django + Django ORM + PostgreSQL

**决策**: 选择 **方案 A — Flask + SQLAlchemy + PostgreSQL**

**理由**:
1. Flask 轻量灵活，模块化架构清晰（Blueprints）
2. SQLAlchemy 成熟稳定，支持类型安全 ORM 和 Alembic 迁移
3. Python 生态在医疗/数据领域丰富（HL7 FHIR、数据分析库）
4. 与前端 TypeScript 虽不同语言，但 AI 对 Python 生成质量同样优秀
5. Flask 可平滑演进到 FastAPI 或微服务架构

**影响**:
- 前后端使用不同语言，类型定义无法直接共享（需维护 API 契约文档）
- Python GIL 在 CPU 密集场景需注意（可用多进程或异步任务）
- 团队需要学习 Flask 的 Blueprint 模块化模式

### ADR-003: 状态管理方案

**状态**: 已采纳
**日期**: [项目启动日期]
**决策者**: 前端负责人

**上下文**:
Taro + React 项目需要状态管理方案。状态包括用户信息、健康数据缓存、UI 状态等。需要考虑小程序的页面栈特性。

**备选方案**:
- A: Zustand（轻量、hooks 原生）
- B: MobX（响应式、Taro 官方推荐）
- C: Redux Toolkit（生态最大、模板代码多）

**决策**: 选择 **方案 A — Zustand**

**理由**:
1. API 极简，AI 生成的 Store 代码可读性高
2. 无 Provider 包裹，在小程序页面栈中使用更自然
3. 内置 persist 中间件，可直接对接小程序 Storage
4. Bundle 体积最小（~1KB）

**影响**:
- 团队需要适应非响应式的状态更新模式
- 复杂的跨 Store 联动需要手动处理（可用 subscribe）

### ADR-004: 模块间通信方案

**状态**: 已采纳
**日期**: [项目启动日期]
**决策者**: 技术负责人

**上下文**:
模块间需要通信（如 HealthData 异常触发 Followup 紧急随访），需要在保持松耦合的前提下实现可靠通信。

**备选方案**:
- A: 直接 Service 注入调用
- B: EventEmitter2 进程内事件总线
- C: Redis Pub/Sub 或消息队列

**决策**: Phase 1 选择 **方案 B — EventEmitter2**，Phase 2 演进到 **方案 C**

**理由**:
1. Phase 1 单体架构下，进程内事件总线足够且零额外依赖
2. EventEmitter2 支持通配符和异步处理
3. 事件发布者不感知消费者，模块边界清晰
4. 演进到微服务时，只需将 EventEmitter 替换为消息队列，业务代码不变

**影响**:
- 进程内事件无持久化，服务重启会丢失未处理事件（Phase 1 可接受）
- 需要约定事件命名规范（module.action 格式）
- 事件处理失败需要有补偿机制（记录失败日志，定时重试）
