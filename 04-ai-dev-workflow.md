# 04 - AI 开发工作流

## 4.1 AI 辅助开发的工作模式

### 三种协作模式

```
模式 A: AI 主导生成（适合标准化代码）
人 ──[需求描述/Prompt]──→ AI ──[生成代码]──→ 人 ──[审查确认]──→ 合入

模式 B: 人机交替迭代（适合复杂业务逻辑）
人 ──[初始需求]──→ AI ──[初稿]──→ 人 ──[修正反馈]──→ AI ──[优化]──→ 人 ──[确认]

模式 C: AI 辅助审查（适合人工编写的代码）
人 ──[编写代码]──→ AI ──[审查建议]──→ 人 ──[采纳/忽略]──→ 合入
```

### 各场景推荐模式

| 开发场景 | 推荐模式 | 推荐工具 | 理由 |
|----------|----------|----------|------|
| CRUD 接口 | A-主导生成 | Claude Code | 标准化程度高，AI 生成质量稳定 |
| DTO/类型定义 | A-主导生成 | Claude Code | 从 SQLAlchemy Schema 直接推导 |
| 复杂业务规则 | B-交替迭代 | Claude Code + Cursor | 需要多轮对话理解业务上下文 |
| UI 页面开发 | B-交替迭代 | Cursor Composer | 可视化调整，多文件协同编辑 |
| 工具函数 | A-主导生成 | Cursor / Copilot | 小粒度，补全即可 |
| Bug 修复 | B-交替迭代 | Claude Code | 需要理解上下文定位问题 |
| 代码重构 | C-辅助审查 | Claude Code | 人主导重构方向，AI 检查遗漏 |
| 测试用例 | A-主导生成 | Claude Code | 从业务代码推导测试场景 |
| 配置文件 | A-主导生成 | Claude Code | CI/CD、Docker 等标准配置 |

## 4.2 Prompt 工程规范

### Prompt 结构模板（CRATE 框架）

```
C - Context（上下文）    项目背景、技术栈、当前模块
R - Role（角色）         AI 扮演的专家角色
A - Action（任务）       具体要做什么
T - Template（模板）     输出格式和代码规范要求
E - Example（示例）      期望输出的参考样例（可选）
```

### 项目级 System Prompt

在 Claude Code 项目中配置 `.claude/settings.json`：

```json
{
  "systemPrompt": "你是慢病管家(ChronicCare)项目的高级全栈开发工程师。\n\n技术栈：\n- 前端: Taro 3.x + React 18 + TypeScript + Zustand + NutUI\n- 后端: Flask 10 + SQLAlchemy + PostgreSQL + Redis\n- 测试: Jest + Testing Library\n\n代码规范：\n1. 所有代码使用 TypeScript strict 模式\n2. 函数和方法必须有明确的返回类型\n3. 使用 ESLint + Prettier 格式化\n4. 组件使用函数式组件 + Hooks\n5. 后端遵循 Flask 模块化架构\n6. 敏感数据字段必须标注 @encrypted\n7. 所有 API 必须有 Swagger 装饰器\n8. 错误处理使用自定义 BusinessException\n\n医疗合规要求：\n- 患者姓名、手机号、身份证必须加密存储\n- 所有数据操作必须有审计日志\n- API 响应中的敏感字段必须脱敏"
}
```

### Prompt 质量评估标准

```
评分维度        1分(差)              3分(中)              5分(优)
──────────────────────────────────────────────────────────────
明确性        需求模糊              基本清楚              无歧义
上下文        无背景信息            部分上下文            完整上下文
约束条件      无约束                部分约束              完整约束
输出格式      未指定                简单指定              详细模板
可复用性      一次性使用            部分可复用            模板化可复用
```

## 4.3 开发流程标准化

### 单个功能的 AI 开发流程

```
Step 1: 需求理解
┌─────────────────────────────────────────────────┐
│ 输入: PRD 中的用户故事 + 验收标准                  │
│ 动作: 用 Claude 将用户故事拆解为技术任务            │
│ 产出: 技术任务清单（含依赖关系）                    │
└─────────────────────────────────────────────────┘
                    │
Step 2: 接口设计    ▼
┌─────────────────────────────────────────────────┐
│ 输入: 技术任务 + 数据模型                          │
│ 动作: 用 Claude Code 生成 API 契约和 DTO           │
│ 产出: Controller 骨架 + DTO 定义 + Swagger 文档    │
└─────────────────────────────────────────────────┘
                    │
Step 3: 业务实现    ▼
┌─────────────────────────────────────────────────┐
│ 输入: 接口定义 + 业务规则描述                       │
│ 动作: Claude Code 生成 Service 层，Cursor 微调     │
│ 产出: 完整的 Service + Repository 实现              │
└─────────────────────────────────────────────────┘
                    │
Step 4: 前端开发    ▼
┌─────────────────────────────────────────────────┐
│ 输入: API 契约 + UI 设计稿/描述                    │
│ 动作: Cursor Composer 生成页面，逐步调整            │
│ 产出: 页面组件 + Service 调用 + Store 集成          │
└─────────────────────────────────────────────────┘
                    │
Step 5: 测试生成    ▼
┌─────────────────────────────────────────────────┐
│ 输入: 业务代码 + 验收标准                          │
│ 动作: Claude Code 生成单元测试和集成测试            │
│ 产出: 测试文件 + 测试覆盖率报告                    │
└─────────────────────────────────────────────────┘
                    │
Step 6: 代码审查    ▼
┌─────────────────────────────────────────────────┐
│ 输入: 完整的功能代码 diff                          │
│ 动作: Claude Code 审查 + 人工审查                  │
│ 产出: 审查意见 + 修复后的代码                       │
└─────────────────────────────────────────────────┘
```

### Prompt 模板：需求拆解

```markdown
# 角色
你是慢病管家项目的技术负责人。

# 用户故事
作为[患者]，我想[记录每次血糖测量值]，以便[医生能查看我的血糖趋势]。

# 验收标准
- Given 患者已登录，When 点击"记录血糖"，Then 显示血糖录入表单
- Given 填写了血糖值和测量时间，When 点击提交，Then 数据保存成功并显示在列表中
- Given 血糖值超出正常范围，When 提交后，Then 显示异常提示并触发医护端预警

# 任务
将此用户故事拆解为可执行的技术任务，每个任务包含：
1. 任务标题
2. 所属层（前端/后端/数据库）
3. 依赖的其他任务
4. 预估复杂度（S/M/L）
5. AI 生成适合度（高/中/低）— 标注该任务是否适合 AI 直接生成
```

## 4.4 代码审查工作流

### AI 审查 + 人工审查双轨制

```
代码提交 PR
    │
    ├──→ [自动] AI 审查（Claude Code）
    │         │
    │         ├── 安全漏洞检查
    │         ├── 医疗合规检查（敏感数据处理）
    │         ├── 代码规范检查
    │         ├── 性能问题检查
    │         └── 输出审查报告
    │
    ├──→ [人工] 团队成员审查
    │         │
    │         ├── 业务逻辑正确性
    │         ├── 架构一致性
    │         └── 可维护性
    │
    └──→ 合并条件: AI 审查通过 + 至少 1 人 Approve
```

### Prompt 模板：代码审查

```markdown
# 角色
你是一名资深代码审查专家，关注安全性、性能和医疗数据合规。

# 审查代码
[粘贴代码 diff 或使用 Claude Code 的 /review 命令]

# 审查维度（按优先级）
1. **安全性** — SQL注入、XSS、敏感数据泄露、权限绕过
2. **医疗合规** — 敏感字段是否加密、审计日志是否完整、脱敏是否正确
3. **正确性** — 业务逻辑是否符合需求、边界条件处理
4. **性能** — N+1查询、不必要的全表扫描、内存泄漏风险
5. **可维护性** — 命名规范、模块边界是否清晰、是否有硬编码

# 输出格式
对每个问题：
- 严重程度: 🔴 必须修复 / 🟡 建议修复 / 🟢 可选优化
- 文件和行号
- 问题描述
- 修复建议（附代码）
```

## 4.5 Git 工作流与 AI 集成

### Feature 开发流程

```bash
# 1. 创建功能分支
git checkout -b feature/blood-glucose-record

# 2. AI 生成代码（Claude Code）
# 在项目根目录执行 claude，描述需求

# 3. 增量提交（每完成一个技术任务提交一次）
git add server/src/modules/health-data/
git commit -m "ai(health-data): 生成血糖记录 CRUD 接口

Tool: Claude Code
Prompt: 基于 HealthRecord 模型生成血糖记录的完整 CRUD
Review: pending"

# 4. AI 审查
# claude review（或在 PR 中自动触发）

# 5. 人工审查后合入
git checkout develop
git merge --no-ff feature/blood-glucose-record
```

### AI 生成代码的版本管理策略

```
原则: AI 生成的代码和人工编写的代码使用相同的质量标准和审查流程

标注规范:
- commit message 使用 ai() 前缀
- 代码文件头部添加 @ai-generated 注释
- PR 描述中记录使用的 Prompt 摘要

追溯能力:
- 通过 git log --grep="ai(" 可筛选所有 AI 生成的提交
- 通过 grep -r "@ai-generated" 可定位所有 AI 生成的文件
- docs/ai-logs/ 保存完整的 Prompt 和产出记录
```
